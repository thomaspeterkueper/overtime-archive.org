# EXT-ECO-OTA-20260714-001 — Schema V1 übernehmen

## Anlass

`kueper-archive-schema` (ECO-ARC-0013) definiert ein gemeinsames
Dokumentarchiv-Schema für KUE und OTA. Laut Umsetzungsplan wendet OTA
dieselben Migrationen an, **nachdem** `kueper.com` sie zuerst importiert und
getestet hat (`EXT-ECO-KUE-20260714-003`).

## Gewünschte Änderung

**Aktualisiert (2026-07-14) — kein eigenes Projekt mehr.** Aus
Kostengründen teilt sich OTA jetzt **dasselbe** Supabase-Projekt wie KUE
(`kue-archive`), getrennt über ein eigenes Postgres-Schema `ota` statt eines
zweiten Projekts. Exakte Anleitung, die dupliziertes Owner-Setup vermeidet:
[`kueper-archive-schema/docs/apply-second-archive-schema.md`](https://github.com/thomaspeterkueper/kueper-archive-schema/blob/main/docs/apply-second-archive-schema.md).

Kurzfassung: `create schema ota; set search_path to ota, public;` → Migrationen
`0001`–`0006` ausführen (landen automatisch in `ota.*`) → RLS mit
`public.is_owner()` referenzieren (nicht `0007`/`0008` erneut ausführen —
das würde Owner-Verwaltung duplizieren) → Seeds aus `seeds/ota/` → Data API
„Exposed schemas" um `ota` ergänzen.

**Bekannte Stolpersteine aus dem KUE-Durchlauf:** pgvector-Extension wird
automatisch in 0006 aktiviert (idempotent, kein erneutes Problem). Details:
`EXT-ECO-KUE-20260714-003.md` in `kueper.com/external-tasks/done/`.

## Begründung

Identisches Schema für beide Archive ist der Kern der Entscheidung
(ECO-ARC-0013 §1) — Abweichungen zwischen KUE- und OTA-Instanz würden das
Ziel unterlaufen.

## Betroffene Repositories

- `overtime-archive.org` (Anwendung der Migrationen)
- `kueper-archive-schema` (Quelle)
- `kueper.com` (muss zuerst abgeschlossen sein)

## Erwartetes Ergebnis

OTA-Supabase-Instanz mit identischem Schema wie KUE, korrektem
`archive_config`-Eintrag (`OTA`, `DOC:OTA:`, Branch `master`) und
OTA-Evidenzprofil.

## Hinweise

Priorität niedrig, da abhängig von `EXT-ECO-KUE-20260714-003` — kein
Zeitdruck, bis KUE abgeschlossen hat.
